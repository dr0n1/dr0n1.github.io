ctf平台搭建和出题指南

ctf平台搭建

平台配置：

CPU 2核 - 内存 4GB - 系统盘 60GB - 带宽 6Mbps

Linux VM-12-3-centos 4.18.0-348.7.1.el8_5.x86_64 #1 SMP Wed Dec 22 13:25:12 UTC 2021 x86_64 x86_64 x86_64 GNU/Linux

Docker version 20.10.14, build a224086

docker-compose version 1.29.2, build 5becea4c

框架选用了最经典的ctfd
动态靶机选用了赵总的whale

但是中间因为版本，依赖的等等问题，经常会报错，所以可以使用如下两种方法，简单又便捷

方法1

使用帆哥的一键部署脚本（centos7和ubuntu20上测试成功）
sh <(curl -s https://raw.githubusercontent.com/pwnthebox/ctfd-auto-deploy/master/install.sh)

注意点：中间需要手动输入三个值
分别是
Http Domain Suffix ：域名/ip
External Http Port : 靶机http映射端口 (这里不能写80，因为80给了nginx，当然也可以自己改下脚本，不使用nginx代理)
Direct Minimum/Maximum Port : 靶机端口开放范围

注：用这个脚本的时候最好挂代理，不然可能跑一天。如果脚本成功运行完毕，那么平台上就无需再次配置，很方便

方法2

使用鲸喵师傅打包好的ctfd和whale插件
先准备好docker环境和docker-compose

创建swarm集群
docker swarm init

加入集群
docker node update --label-add='name=linux-1' $(docker node ls -q)

拉取镜像
git clone https://github.com/VaalaCat/CTFd.git --depth=1
git clone https://github.com/VaalaCat/ctfd-whale.git

修改token
两个文件的token值修改成一样的，如果用到了域名记得修改subdomain_host
vim CTFd/conf/frp/frps.ini
vim CTFd/conf/frp/frpc.ini

安装插件（在CTFd目录下运行）
git submodule update --init

启动（在CTFd目录下运行）
docker-compose up -d

访问http://xx.xx.xx.xx:9124 (默认是9124，想要80可以修改docker-compose.yml)

平台配置

右上角的插件中进入 Whale 设置，其中，左侧 Docker/Limits/Challenges 菜单栏根据需求自行更改， Frp 菜单栏中的内容需要更改一下几个选项

Http Domain Suffix node.lewiserii.top 这里填写使用http方式访问靶机的泛解析域名
Http Port 8080 这里填写frps中的vhost_http_port，该端口为http方式靶机访问的端口
Direct IP Address xxx.xxx.xxx.xxx 这里填写服务器ip，用于显示Direct方式访问的题目的IP
Direct Minimum Port 9000 这里填写用于动态靶机Direct方式的开始端口
Direct Maximum Port 9200 这里填写结束端口

优点：非常稳定，基本都能一次成功

pwn题部署

先展示一下目录结构

首先先出题，出完后的pwn2.c文件编译成pwn2，并和flag文件放入bin下
注意flag文件中的flag格式要与ctfd中配置的格式一致

然后编辑Dockerfile文件，用来构建镜像

再编辑start.sh文件用于启动环境

既然启动了xinetd服务，那么肯定要有配置文件
编辑ctf.xinetd，这个文件是有固定格式的，可以参考下面的

当ctf.xinetd,Dockerfile,start.sh三个文件和题目pwn2都配置完后就可以开始构建和运行了

构建（在配置文件所在的目录）
docker build -t vecctf_pwn2 . # vecctf_pwn2是构建完的镜像的名字

运行
运行前可以使用docker images命令查看是否生成了镜像
docker run -itd -p 9999:9999 vecctf_pwn2 /start.sh # 注意端口和镜像要对应

现在使用exp试试能不能打通（对应上面目录结构里的pwn2.py）

能打通就说明镜像构建成功了

然后在平台部署

如果平台和镜像不在同一台主机上可以先push在pull下来

Challenge Types选择dynamic_docker

选项 内容
Name 挑战名称
Category 挑战类别
Message 简要介绍挑战
Docker Image 部署的docker镜像名称
Frp Redirect Type http是映射域名，direct是映射ip
Frp Redirect Port 重定向端口
Docker Container Memory Limit 容器内存
Docker Container CPU Limit 容器cpu
Initial Value 初始分数
Decay Limit 挑战达到其最小值之前的解决数量
Minimum Value 最低分数
Score Type 静态分数&动态分数

提交后的flag选项不用填，state选择显示，上传pwn题附件

一切就绪后可以试着开几次靶机，看看flag和端口是不是动态的

go靶机

从简单的开发与学习过程中可以感觉到golang的确非常安全，很多问题由于作为强类型的静态语言，从编译过程就能防止不安全的操作。因此大部分漏洞产生还是归结于依赖库（通常已经发展为CVE或者issue）以及开发者自身的操作漏洞。[1]

go靶机平时基本见不到，比赛中也非常少见，最近正好在学golang，就试着做一下靶机

在dockerhub上搜了一下，似乎没有合适的基础镜像，刚开始尝试用golang和ubuntu来做基础镜像，但是最后都太大了，一个G起步，不得已用alpine做了一个出来

go的题目和java一样分两种情况，一种是编译好的二进制文件，这种搭建就比较方便了，可以采用多阶段构建来缩小容量，具体可以参考雪殇师傅的文章[2],另一种就比较麻烦了，需要用go run来运行的题目，比如羊城杯的Checkin_Go

这里构建就不演示了，与php靶机搭建差不多，主要就是go mod download && go mod verify和go run .

还有一点要注意，平台上题的时候需要调整Docker Container Memory Limit，不然会报go build github.com/ugorji/go/codec: /usr/local/go/pkg/tool/linux_amd64/compile: signal: killed，问题应该出在镜像alpine上

带数据库的靶机

像一般的sql注入类和网站类题目是需要用数据库存储数据的，所以需要基础镜像中包含一个数据库

推荐镜像1：andreisamuilik/php5.5.9-apache2.4-mysql5.5
推荐镜像2：lewiserii/php7-apache2-mariadb5

部署时在sh文件中导入你需要的数据即可：mysql -uroot -proot < /tmp/db.sql
